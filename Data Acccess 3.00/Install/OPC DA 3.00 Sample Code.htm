Overview

This update is built against the latest version of the .NET API assemblies.

This release of the Data Access samples now runs on x64 systems and includes binaries and merge modules with authenticode signatures. The file version attribute in the executables has been incremented to *.*.100.0 to ensure that all binaries released together have a common portion of the version number. This release of all samples built with the .NET API is collectively called "Build 100". Any problems reports related to the binaries in this distribution should reference this build number. The product version field in the binaries has not been incremented since the APIs have not changed.

The .NET COM interop assemblies are built against the 32-bit versions of the proxy/stubs. This means that any .NET application running on x64 systems must be compiled and run as a 32-bit application. Developers using the RCWs can ensure their applications run correctly if they set the platform target to x86 for any .NET executable that directly or indirectly uses the RCWs. The platform target setting can be found by opening 'Project Properties' and selecting the 'Build' tab.

The OPC Data Access Sample Code consists of C# client and server applications and a C++ server application. These applications implement the Data Access Custom Interface Specification 3.00 and 2.05a. These samples are intended to demonstrate the correct operation of the specification and to serve as starting point for vendors who wish to develop their own implementations.

The sample client is written completely with .NET technologies; as a result, it also demonstrates how to build .NET applications that interoperate with existing OPC compliant COM servers. Part of the sample client includes a set of classes and interfaces that expose a common native .NET interface for both COM-DA and XML-DA servers. This .NET API will be maintained a separate component that vendors can use as a basis for their own .NET based applications.

DCOM configuration with Windows XP SP2

Microsoft has provided some sample APIs which can be used to programmatically configure DCOM permissions under Windows XP SP2 and later systems. The sample code now includes these APIs as part of the 'Sample Utility Classes' project. The APIs are documented in the 'DCOM Config Readme.htm' file which is also part of that project.

Server Implementation in .NET

The sample code includes a server implemented in C#. This server is implemented using the .NET API server interface which is then registered as an in process COM server. However, the .NET COM interoperability layer has a number of limitations that make it impossible to develop a compliant DA server purely in .NET. As a result, the sample code also includes a wrapper written in C++ that can host any implementation of the DA interfaces in .NET. This wrapper is designed to be generic and supports any .NET in-process server that implements the DA COM interfaces. The XML configuration file for the wrapper specifies the ProgID of the .NET server it loads.

Note that the .NET server implementation uses same code base as the XML-DA sample server, as result, this sample also demonstrates how to write a DA server in .NET that supports both the COM DA and XML DA specifications.

Complex Data

This version (3.00.2.02) of the sample clients and servers have been modified to support the draft complex data specification. Complex data related features are accessible in the sample client user interface via context menus or buttons used to view or edit an item value. If the item is a complex item (i.e. its defines the complex data properties described in the specification) then a complex item dialog will appear. This dialog displays the complex data in its raw form (binary or text) as well as a tree view. The type description information provided by the server is also accessible via this dialog.

The underlying items exposed by the sample servers are binary values which may be converted to XML by server if requested by the client. The client application is able to display a structured view of the complex data by fetching the OPC Binary type description from the server and using this to parse the buffer returned by the server.

System Requirements

Operating Systems

All source code provided by the OPC Foundation is only supported on Windows XP SP2 and Windows Vista. That said; the source code has been designed to be portable to other operating systems. For this reason vendors are encouraged to report fixes required to port the samples to other operating systems so they can be incorporated into the baseline and to improve the portability of future sample code.

The samples required the .NET Framework version 2.0. The .NET Framework can be downloaded from here.

All C++/COM applications are compiled as UNICODE applications and will not run on Windows 98 or Windows ME unless they are recompiled.

The installers come with a setup.exe file. This is the file that must be used to install under Windows Vista. Double clicking on the MSI will cause an error.

Usage

Server

The sample server opens and reads the XML file that must be located in the same directory as itself. The XML file must have the same base name as the server (i.e. 'OpcDaServer'). When executed, the server parses the file a constructs its address space from the items defined within it. The file defines all standard DA properties for each item as well as some simulation parameters (period and sampling rate). Only items that have a value for the period specified will change dynamically. The waveform depends on the data type: float point types are sinusoids; integer types are a linear ramps and enumerated types cycle through the set of possible values.

Server Wrapper

Connecting to a .NET implementation of a DA COM server requires two distinct registration steps. First, the .NET assembly must be registered as an in-process COM server. This can be done programmatically or with the OpcRegSvr.exe command line utility. This utility behaves like the regsvr32 utility. Second, the wrapper configuration file (OpcDaWrapper.config.xml) must be updated with the ProgID of the .NET server. It is also possible to change the ProgID and CLSID of the wrapper itself in this configuration file. The ProgID of the wrapper is important because it is this ProgID which the COM clients must use to connect to the server. The format of the configuration file is intended to be self documenting.

Client

The sample client can be started from the programs menu. By default, it will connect to the sample server when the connect button is clicked. The sample client comes with a standard configuration file that creates three groups on the sample server (described below). The client can connect to any server that supports DA 2.0 or 3.0, however, DA 3.0 specific functionality is disabled if the client detects that the server does not support DA 3.0. The connect toolbar button or menu item allows the client to browse the local network for OPC servers on other machines. While navigating the client always try to right click or double click to access functionality (such as viewing the elements in an array value).

Basic Types

This group demonstrates reading and writing of all supported data types, including support for server side data conversions between types as defined in Data Access 2.05 and the writing of quality and timestamp. Right-click on the left tree view to access the read/write functionality. None of the items in this group change, as a result, the server sends a keep alive update every 5s (the keep alive rate can be changed in the Edit Group dialog).

Deadbands

This group demonstrates the deadband specified at the item level. The group has the same item added twice, except one of the items has a deadband of 0% that overrides the group level deadband of 15%. The item changes in sinusoidal pattern so the updates for the item with the group deadband of 15% are not reported when the value is near the peaks of the sinusoid.

Buffering

This group demonstrates the item level sampling rate and buffering. The group has an update rate of 5s and two items that change once every second. The items have buffering enabled, however, one item has a sampling rate of 2s and the other has a sampling rate of 3s. Right click on the right side view and click enable buffering. This displays all values returned in a single update (shown in red). The server should deliver an uninterrupted series of 2s samples in blocks of 2 or 3 for the item with the 2s sampling rate. The item with the 3s sampling rate has a series of 3s samples in blocks of 1 or 2

Enumerations

This group demonstrates the use of enumerated items where the set of valid string values is stored in the EU Info property. The four items receive updates from the server both as strings and integer values.

Dynamic Errors

This group demonstrates how dynamic error codes are reported during data change callbacks. It is possible to toggle between display of the localized error texts and the standard error code by right clicking on right pane list. Note, that data change callbacks are only sent if the error state changes.

Complex Data - Binary

This group demonstrates the subscriptions to several complex data items described with the OPC Binary type description system. The 'View' menu option in the popup menu will display dialog that shows the item value in raw an structured format. It also shows the type descriptions provided by the server. These items do not change but it is possible to modify the complex value and write values to the server.

Complex Data - XML

This group demonstrates the subscriptions to several complex data items described with the XML Schema type description system. The 'View' menu option in the popup menu will display dialog that shows the item value as an XML document. It also shows the type descriptions provided by the server. These items do not change. 

The items in this group as represent type conversions for each of the items in the OPC Binary group above. It is possible to enable/disable type conversions via the 'Edit Item' dialog which is accessible when clicking on an item in the tree view on the left.

These items also support Complex Data Filters. These filters may also be entered via the 'Edit Item' dialog. The syntax is any valid XPath that can be applied to the value. For example, the filter '//Uint' when applied to the 'Integers' item will cause the server to only return a value containing elements named 'UInt'.

Complex Data - Dynamic

This group contains one item in binary and XML formats that changes dynamically. The data updates can be monitored in the 'View Complex Data' dialog that is accessible via the 'View' menu in the right hand list view.
